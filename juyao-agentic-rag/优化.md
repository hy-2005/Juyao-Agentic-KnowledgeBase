# RAG 检索效果优化任务清单

> 用法：每完成一项把 `[ ]` 改成 `[x]`；新增任务直接追加到对应分组末尾。

## 0. 触发本次优化的现场

**Query**：`若系统于 2024 年 2 月 25 日通过验收，但甲方在 3 月 5 日发现 [问题]，3 月 10 日修复完成。尾款最迟应在哪一天支付？并说明依据合同哪两条条款推导得出。`

**应有证据（在知识库里、但没被送到 LLM）**：

- `chunk:2`（合同 2.2 付款方式 / 含「尾款：产品安装调试完成并稳定运行 30 天后」）—— **核心证据**
- `chunk:6`（合同 6.x 违约责任 / 含 6.2 质量问题处理）—— 辅助证据

**实际送到 LLM 的 3 条**（全是噪声）：

- `chunk:0`（合同抬头：买卖双方信息）
- `chunk:4`（4.x 安装时间 + 5.1 质保期限）
- `chunk:5`（5.x 质保 + 5.3 响应时间）

**失败链条总结**：

| 关键 chunk | 向量名次 | ES 名次 | RRF 排名 | rerank 后 |
|---|---|---|---|---|
| `chunk:2`（核心证据） | **15（最后）** | 3 | 8（压线） | **被淘汰** |
| `chunk:6`（辅助证据） | 12 | 2 | 5 | **被淘汰** |
| `chunk:5`（无关） | 1 | 1 | 1 | 留下 |

**结论**：知识库不缺答案，是检索/排序流程把答案过滤掉了。

## 1. 已识别的卡点（按严重度排）

- [ ] **卡点 A（致命）**：DashScope `gte-rerank-v2` 把核心证据 `chunk:2/6` 直接淘汰；rerank 是单模型单次打分，错就错到底，没有兜底。
- [ ] **卡点 B**：向量检索把 `chunk:2` 排到第 15 位（所有 chunk 相似度都挤在 0.81~0.85，几乎没分辨力）；`min_relevance_score=0.35` 在当前 embedding 模型下完全失效。
- [ ] **卡点 C**：BM25 把含「尾款」的 `chunk:2`（39 分）排在含「验收/响应时间」的 `chunk:5`（53 分）后面；query 里多个常见词稀释了「尾款」的权重。
- [ ] **卡点 D**：切分阶段没有章节结构感知（`splitter.py` 提示词没有要求按 `第X条 / X.Y` 硬断），元数据里也没有 `section_title / clause_no`；rerank 看不到强语义锚点。
- [ ] **卡点 E**：管道宽度逐级砍半（`top_k=15 → rrf_top_n=8 → rerank_top_n=3`），关键证据一旦"压线"就毫无冗余。
- [ ] **卡点 F**：query 没有改写/扩展；用户问的"算日期"型问题，字面上不含「安装调试 / 稳定运行」，向量和 BM25 都没机会命中。
- [ ] **卡点 G**：没有「邻居 chunk 召回 / 同 source 章节扩展」；即便核心 chunk 命中，相邻条款也不会一起带进上下文。

## 1.5 卡点 A 深度分析与常规方案定位（讨论补充）

> 这一节回答两个问题：
> 1. 卡点 A（rerank 单点决策）单独修能不能救回本题？
> 2. 这种"算日期/推理型"query 是不是常规检索方案做不了？
>
> 结论先放：**单独修卡点 A 救不回本题；但组合"Query 改写 + rerank 兜底"完全在常规方案能力范围内**。

### a) 卡点 A 的本质——不是"抽错"，是"在弱信号上做单点决策"

把 RRF 8 条候选的真实分数 + rerank 结果 + 真实相关性摆一起：

| RRF 排名 | chunk | RRF 分 | rerank 输出 | 真实相关性 |
|---|---|---|---|---|
| 1 | chunk:5（5.x 质保 / 5.3 响应时间） | 0.0320 | top1 ✅ | ❌ 无关 |
| 2 | chunk:4（4.2 安装 / 5.1 质保） | 0.0320 | top2 ✅ | ❌ 无关 |
| 3 | chunk:3（3.x 交货） | 0.0308 | 淘汰 | ❌ 无关 |
| 4 | chunk:8（9.x 不可抗力） | 0.0302 | 淘汰 | ❌ 无关 |
| 5 | **chunk:6**（6.x 违约责任） | 0.0300 | **淘汰** | ✅ **辅助证据** |
| 6 | chunk:0（合同抬头） | 0.0296 | top3 ✅ | ❌ 无关 |
| 7 | chunk:9（11.x 变更） | 0.0294 | 淘汰 | ❌ 无关 |
| 8 | **chunk:2**（**2.2 付款方式 / 尾款**） | 0.0292 | **淘汰** | ✅ **核心证据** |

三个观察：

1. **rerank 不是小错，是全错**：top 3 里没一条真相关；说明 rerank 不是"边缘判错"，是在 8 条上下文几乎没区分度的情况下整体翻车。
2. **RRF 自己也没把 chunk:2 排前**：核心证据只排第 8（压线进入 rerank 候选），辅助证据 chunk:6 排第 5。
3. **8 条 RRF 分数差距才 ~10%**（0.0292 ~ 0.0320）：两路检索给出的"信号差"本身就微弱，rerank 在弱信号上做单点决策几乎等于抛硬币。

> **本质判断**：卡点 A 的根因不是"rerank 这次抽中了错的"，而是"rerank 是一个高方差单点决策模块、它的输入候选区分度本来就低"。

### b) 四个候选 rerank 兜底方向的取舍

| 方向 | 思路 | 能救本案？ | 实现成本 | 副作用 |
|---|---|---|---|---|
| **A. rerank × RRF 加权融合** | `final = α·rerank + (1-α)·rrf` | ⚠️ **救不回**：chunk:2 在 RRF 也只第 8，加权后仍然排不到前 3 | 低 | α 调参 |
| **B. RRF + rerank 取并集** | rerank top_n ∪ RRF top_K，最终送 LLM 5~7 条 | ⚠️ **部分救回**：K=5 能救 chunk:6（RRF 第 5），救不到 chunk:2（RRF 第 8）；K=8 才能全救 | 极低 | 上下文变长 2~5 条 |
| **C. rerank 输入加锚点** | 给 rerank 看的 page_content 前面拼 `[来源/章节]` | 🤔 **不一定救得到本题**，但能改善"开头被截断/无标题"的系统性偏差 | 低（不依赖 Task 3 也可拼 `source_name`） | 可能干扰正文判断 |
| **D. 分歧检测 + LLM-as-rerank 兜底** | 当 rerank 与 RRF 交集 ≤ 1 时，调一次 GLM 让它从 RRF 前 8 里挑 3 条 | ✅ **大概率救回**：LLM 看到"尾款/质量问题"两条款 vs 用户问题，能判对 | 中（多 1 次 LLM 调用） | 延迟 / 成本 +1 次 |

> **判断**：A、B、C 都建立在"两路检索信号本身够强"这个前提上，对本案这种"RRF 自己也错"的极端场景天花板很低。**只有 D 能在不依赖切分重做（Task 3）的前提下，把语义判断能力实质性提升一档**——但 D 的最佳触发条件，是先有"分歧度"信号，所以仍然要先把 A+B 做出来。

### c) 这道 query 是"召回+重排死区"——为什么单独修卡点 A 不够

把用户的 query 和真实答案条款的字面交集摆一起：

| | 字面内容 |
|---|---|
| **Query** | 验收、2 月 25 日、3 月 5 日发现、3 月 10 日修复、**尾款**、最迟、哪一天支付 |
| **答案 chunk:2** | 预付款、进度款、**尾款**、安装调试完成、**稳定运行 30 天**、合同总价 30%、人民币 1,275,000 元 |

字面交集只有"尾款"两个字。其他全是**类型不同**的词：

- query 是**时间事件 + 推理动词**（验收/发现/修复/最迟/推导）
- 答案是**条款实体 + 触发条件**（安装调试/稳定运行 30 天/合同总价/付款比例）

让任何"单次相似度计算 + 单次 rerank"的方案直接在两段文本之间找强匹配，本质上是在比较两个**主题空间**的词。**这是召回+重排架构的天然短板，调参救不了**。

### d) 常规检索方案的"工具箱地图"——我们覆盖了哪几层

| 层级 | 常用手段 | 我们当前 | 缺什么 |
|---|---|---|---|
| **召回层** | 向量、BM25、混合并行 | ✅ 向量 + ES + RRF | — |
| **重排层** | Cross-encoder rerank | ✅ DashScope `gte-rerank-v2` | 兜底 / 融合 |
| **Query 侧** | 改写、HyDE、Sub-query 分解、Step-back | ❌ 没有 | 全部 |
| **后检索侧** | 邻居扩展、章节扩展、Self-RAG 反思补检索 | ❌ 没有 | 全部 |
| **结构感知** | 章节标题作锚点、metadata 加权 | ❌ 没有 | 全部 |

**我们现在只用了"工具箱里最基础的两层"**（召回 + 重排），剩下三层完全没用。这道合同题恰好就是"必须靠后三层才能解决"的典型——所以失败不奇怪，但也**不代表常规方案做不了**。

工业界对"推理型/计算型 query"的标准动作就是**先改写、再检索**，让 LLM 把 query 拆成事实查找型 sub-query：

```
原 query → 拆成 sub-query：
1. 合同里关于尾款支付时间的约定是什么？
2. 合同里关于稳定运行多少天后支付尾款的条款？
3. 合同里关于产品质量问题处理 / 修复期间是否顺延付款的条款？
```

把 sub-query #1 喂给当前系统，**chunk:2（"尾款：产品安装调试完成并稳定运行 30 天后..."）几乎一定能排进 top 1**——因为字面就直接匹配。这个手法叫 **Query Rewriting / Multi-Query Retrieval**，是 LangChain / LlamaIndex 都内置的标准组件，**就是常规方案**。

类似的还有：HyDE（让 LLM 先"假装"写一段答案，用假答案 embedding 去检索）、Step-back Prompting（先问"要回答这个需要什么先决知识"）、Self-RAG（第一次检索后让 LLM 判断证据够不够，不够再发起新检索）——**这些都是 2023~2024 年间已经定型的常规手段**，只是我们还没接进来。

### e) 对 Task 优先级的反推

基于以上分析：

- 🔴 **Task 1（Query 改写）** 是把这道题从"召回+重排死区"挪进"召回+重排能搞定的范围"的**关键一步**——必须先做。
- 🔴 **Task 2（rerank 兜底）** 修的是"边缘判错"，对 80% 的常规 query 有效；但本案太极端（rerank 全错 + RRF 自己也错），单独做救不回，**必须和 Task 1 配合**。
- 🟡 **Task 3 / Task 4 / Task 5** 是"结构感知 + 后检索侧"的手段，是工具箱的另外两层；做完之后即使不开 Task 1 也能扛住一部分推理型 query。
- 🔵 **Task 10 LLM-as-rerank** 是 D 方案的实现，**先有 Task 2 的"分歧度"信号才好触发**——所以放在 Task 2 之后。

> **一句话总结**：我们现在不是"常规方案做不到"，而是"常规方案的工具箱只用了 30%"。Task 1 + Task 2 做完工具箱用到 60%，本题就能答对，绝大多数同类 query 也都能搞定。剩下 40% 留给特别难、需要多轮交互的 query。

## 2. 优化任务清单（按性价比从高到低）

### 🔴 P0 · 立竿见影（强烈建议先做）

#### Task 1 · Query 改写 / Multi-Query 检索
- [ ] **目标**：让"算日期"型问题也能召回到含触发条件的条款。
- [ ] **思路**：在 `search_context` 入口前加一步 LLM query 改写，把原 query 拆成 3~5 条事实查找型 sub-query，并行检索后合并去重再走 RRF。
- [ ] **改动点**：
  - [ ] 新增 `rag_core/rag/retrieval/query_rewrite.py`，封装 LLM 调用 + 提示词。
  - [ ] `retriever.search_context` 在两路检索前，先生成 sub-query 列表；每条 sub-query 各跑一次双路 top_k。
  - [ ] 多路结果按 `chunk_id` 去重，名次取「最小 rank」（即在任一 sub-query 里的最佳名次）参与 RRF。
  - [ ] 在 `config.py` 新增 `query_rewrite_enabled / query_rewrite_max_subqueries` 两个开关。
- [ ] **验收**：用本次合同问答 query 复跑，`chunk:2` 必须出现在最终 LLM 上下文里。
- [ ] **风险**：增加 1 次 LLM 调用延迟；需对 rewrite 失败做 fallback（回退到原 query）。

#### Task 2 · rerank 兜底策略（组合方案：取并集 + 加权融合 + 分歧检测）
- [ ] **目标**：防止 rerank 单点决策一票否决核心证据；对应 1.5 节方向 A + B + 触发 D 的前置信号。
- [ ] **基础层（必做）：rerank top_n 与 RRF top_K 取并集**
  - [ ] 最终送 LLM = `rerank_topN ∪ RRF_topK`（去重后预计 5~7 条）
  - [ ] `config.py` 新增 `final_union_rrf_topk`（默认 3，0 表示关闭）。
  - [ ] 副产品：`rerank_top_n` 默认 3 → 5（与 Task 6 联动）。
- [ ] **加权层（必做）：并集内的最终顺序由 α·rerank + (1-α)·rrf 决定**
  - [ ] `_rerank_after_rrf` 保留 rerank 返回的 `relevance_score`，与归一化后的 `rrf_score` 加权求和。
  - [ ] 归一化策略：两路分数各自 min-max 归一化到 [0,1]；某 chunk 在 rerank 输出里没出现时按 0 处理（或回退到 RRF 最低分）。
  - [ ] `config.py` 新增 `rerank_blend_alpha`（默认 0.5；α=1 等同纯 rerank，α=0 等同纯 RRF）。
- [ ] **可观测性层（必做）：分歧度日志**
  - [ ] 计算 `Jaccard(rerank_topN, RRF_topN)` 并打印；
  - [ ] 当交集 ≤ 1 或 RRF 前 3 全被 rerank 踢出时打 WARN（这是 Task 10 LLM 兜底的触发信号）。
- [ ] **验收**：本次 query 复跑时，
  - [ ] `chunk:6`（RRF 第 5）必须出现在 LLM 上下文（取并集即可救回）；
  - [ ] `chunk:2`（RRF 第 8）单靠本任务**预期救不回**——必须配合 Task 1 才能进入 LLM 上下文，本任务不强制。
- [ ] **风险**：上下文条数从 3 → 5~7，token 成本上升约 70%；需观察对 GLM 答题质量的影响。
- [ ] **不在本任务范围**：当分歧度 WARN 触发后调 LLM 重排——那是 Task 10。

### 🟡 P1 · 系统性收益（建议第二批做）

#### Task 3 · 切分识别合同/规章章节编号
- [ ] **目标**：让"第 X 条 / X.Y"成为硬切分边界，并把章节标题写入 metadata。
- [ ] **改动点**：
  - [ ] `splitter.py` 主路径提示词追加："遇到 `第X条`、`X.Y`、`X.Y.Z` 等条款编号必须断开"。
  - [ ] 新增 `_extract_section_title(content, span)`：在 chunk 内/上方就近回溯条款标题（如 "2.2 付款方式"）。
  - [ ] `enrich_chunk_metadata` 增加 `section_title` / `clause_no` 字段。
  - [ ] `elasticsearch_store` 的 mapping 把 `section_title` 作为独立字段、加权检索（如 `section_title^3`）。
- [ ] **验收**：重新入库后，问"尾款"应在 BM25 里直接命中含 `section_title=2.2 付款方式` 的 chunk 排第 1。
- [ ] **依赖**：需要重新跑一次 ingestion；ES mapping 要清空重建。

#### Task 4 · 邻居 chunk 召回 / 同 source 章节扩展
- [ ] **目标**：核心 chunk 命中时自动把相邻条款也带进上下文。
- [ ] **改动点**：
  - [ ] `search_context` 在 rerank 输出后，对每条 chunk 取同 `source_doc_id` 的 `chunk_index ± 1` 邻居，去重后追加到 documents 末尾（不进 LLM 引用列表，仅作上下文）。
  - [ ] `config.py` 新增 `neighbor_expand_window`（默认 1，0 表示关闭）。
- [ ] **验收**：`chunk:2` 命中后，`chunk:3`（2.3 付款条件 / 3.x 交货）应一起出现在上下文。
- [ ] **风险**：上下文变长、token 成本上升；需控制窗口大小。

#### Task 5 · 把 `section_title` 拼到 chunk 文本前再做 rerank
- [ ] **目标**：给 rerank 模型一个强语义锚点，缓解"开头是金额尾巴"导致的低分。
- [ ] **依赖**：Task 3 完成后才有 `section_title`。
- [ ] **改动点**：
  - [ ] `_rerank_after_rrf` 在构造 `documents_payload` 时，前缀拼上 `[第二条 价格与支付方式 / 2.2 付款方式]\n` 这样的标题。
  - [ ] 仅在送给 rerank 时拼，**不影响最终送 LLM 的 page_content**（避免重复）。
- [ ] **验收**：本次 query 在不开 Task 1/2 的情况下，rerank 也能把 `chunk:2` 排进前 3。

### 🟢 P2 · 小补丁 / 调参（随时可做）

#### Task 6 · 放宽管道宽度
- [ ] `rrf_top_n`：8 → 12
- [ ] `rerank_top_n`：3 → 5
- [ ] **验收**：本次 query 复跑时 `chunk:2` 至少能出现在 LLM 上下文中（哪怕排第 5）。

#### Task 7 · `min_relevance_score` 改成相对阈值
- [ ] **现状**：固定 0.35，但 `mxbai-embed-large` 实际相似度普遍 > 0.8，过滤完全失效。
- [ ] **思路**：改成 `top1 - delta`（如 `delta=0.05`）或分位数（保留 top 60%）。
- [ ] **改动点**：`retriever._vector_similarity_topk` 之后过滤逻辑改写；`config.py` 新增 `min_relevance_mode`（fixed / relative_delta / quantile）。

### 🔵 P3 · 长期收益（要重新入库 / 大改）

#### Task 8 · 升级 embedding 模型
- [ ] 候选：`bge-large-zh-v1.5` / `bce-embedding-base_v1` / `conan-embedding-v1`。
- [ ] **验收标准**：在合同/技术报告混合知识库上，前述 query 的 `chunk:2` 向量名次从 15 → 进入前 5。
- [ ] **代价**：需要重新生成所有 embedding 入库。

#### Task 9 · `chunk_size` 调整与对比实验
- [ ] 试 300 / 500 / 800 三档，看合同条款类文档的召回 P@3 / Recall@10。
- [ ] **代价**：每次都要重新切分入库。

#### Task 10 · 引入 LLM-as-rerank 备选通道
- [ ] 当 RRF 与 `gte-rerank-v2` 分歧极大（如 RRF 前 3 全被踢出）时，调用一次 GLM 自己判断哪些 chunk 与问题相关。
- [ ] **代价**：成本最高的 rerank 路径，仅作 fallback，不做主路径。
