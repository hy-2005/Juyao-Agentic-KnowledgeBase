## 本阶段技术栈

| 类别 | 选型 |
|------|------|
| 向量检索 | **Qdrant**（同阶段 1 嵌入模型） |
| 关键词检索 | **Elasticsearch（ES）**；与向量 **并行** 检索 |
| 结果融合 | **RRF** 或加权合并（实现层自定，行为需可配置） |
| Query 侧 | 改写、**多 Query 扩展**（Demo 优先）；HyDE / Step-back 按需 |
| 重排 | **`jina-reranker-v2`**（Jina API 或兼容调用，以实现为准） |
| 评测 | **Hit@K**、**MRR**、**RAGAs** |

## 开发目标

1. 提升召回率与 Top-K 质量，为后续「是否需要走图谱」的路由提供稳定语义基础。
2. **Query 改写**与**多 Query 扩展**（Demo 优先）；HyDE、Step-back 等列入 backlog 按需启用。
3. **混合检索**：Qdrant 向量检索 + Elasticsearch 关键词检索并行，结果融合（如 RRF 或加权策略）。
4. **重排**：对候选 chunk 做 cross-encoder 或 API 重排，再送入 LLM。
5. 建立离线评测：**Hit@K**、**MRR**，并接入 **RAGAs** 跑通流程。

## 交付物

- 可配置的检索管线开关（是否扩写、是否混合、是否重排）。
- 与阶段 1 同测试集的**前后对比**记录（表格或简短报告）。
- 一版**默认超参数**冻结为后续阶段基线。

## 验收要点

- 黄金集上 Hit@K / MRR 或 RAGAs 核心指标相对阶段 1 有可见提升（或明确记录失败原因与下一动作）。
- 混合检索在「专有名词、药品名」类 query 上不明显弱于纯向量。
