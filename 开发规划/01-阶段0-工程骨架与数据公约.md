## 开发目标

1. 建立可重复执行的「文档 → 处理 → 可检索存储」最小流水线。
2. 约定 **chunk 全局唯一 ID**，保证后续向量库与 Neo4j 能指向**同一段文本**，避免第三方 GraphRAG 自拆 chunk 后与向量不一致。
3. **Chunk 切分（当前约定）**：**Ollama** 本地 **`qwen2:1.5b`** 做语义切割；**兜底**为相邻 chunk **前后段重叠覆盖**，减少边界信息丢失。切分模型与后续 embedding/问答模型分离配置。
4. 打通配置与模型调用（Python + LangChain 等），为后续阶段预留接口；本阶段需能拉起 Ollama 并完成一次端到端切分验证。

## 交付物

- 项目目录结构、依赖与配置说明（**`juyao-agentic-rag/requirements.txt`** + `src/rag_core/config.py`）。
- 统一 **chunk 元数据字段**：至少包含 `chunk_id`、`source_doc_id`、`offset` 或序号、可选 `hash`；若启用重叠，记录 **`overlap` 范围或与相邻 chunk 的关联**，便于增量更新时成批失效。
- **语义切分**：基于 `qwen2:1.5b` 的切割脚本或管线说明（含 Ollama 地址与模型名配置项）。
- 最小数据集跑通：1～2 份清洗后的文本/PDF 经上述策略切块后入库（向量库为本地 Qdrant，默认 `http://localhost:6333`）。
- 配置分层：API Key、各模型名（切分 / embedding / 生成）、向量维度等与代码分离。

## 验收要点

- 同一批文档可重复入库，**chunk_id 稳定**（同一文本同一规则下 ID 不变）。
- 语义切分 + 重叠策略跑通，抽查相邻边界 topic 不被「硬切断」到不可检索（允许用重叠段补救）。
- 团队内有一份简短「数据公约」说明：谁生成 chunk_id、哪些模块必须读写该字段。

## 依赖与风险

- **依赖**：无前置阶段。
- **风险**：若此处 ID 不统一，阶段 3～4 图谱与向量会对不齐，返工成本高。
